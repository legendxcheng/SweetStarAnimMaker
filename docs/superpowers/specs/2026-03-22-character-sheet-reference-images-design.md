# Character Sheet Reference Images Design Spec

**Date:** 2026-03-22
**Status:** Approved
**Scope:** `apps/api`, `apps/studio`, `apps/worker`, `packages/core`, `packages/services`, `packages/shared`

---

## Summary

Extend the `character_sheets` stage so each character can own a persistent set of local reference images.

These reference images are managed inside the existing character-detail workflow. Users can add and remove reference images per character from the Studio UI. When a character image is generated or regenerated, the system automatically includes that character's current reference-image set if one exists. If no reference images exist, generation continues as the current prompt-only flow.

This feature reuses the existing VectorEngine reference-image upload path in the worker. The missing work is character-level persistence, API surface, task-input wiring, and Studio interaction.

## Goals

- Let users attach reference images to an individual character in the `角色设定` phase
- Persist reference images inside project storage so they survive refreshes and future regenerations
- Keep generation optional: no reference images must still produce a valid character sheet
- Reuse the existing worker-side reference-image uploader and VectorEngine `image[]` integration
- Keep the first release limited to preview and delete, without extra metadata editing

## Out Of Scope

- Project-level shared reference libraries
- Sorting reference images
- Reference-image notes, tags, or captions
- Browser-side direct upload to the public image host
- Uploading generated output images to a public host
- Reworking the entire character-sheet review layout or adding generated-image preview in this release

## User Workflow

For each character in the `角色设定工作区`:

1. Open a character detail panel.
2. Add one or more local reference images from the browser.
3. The browser uploads those files to the API.
4. The API saves them into project storage under the current character's directory and updates a manifest.
5. The UI immediately shows thumbnail previews and delete actions for those reference images.
6. When the user clicks `重新生成`, the system automatically includes the current reference-image set for that character.
7. If the character has no reference images, the same button still triggers a valid prompt-only generation request.

The batch-generation path follows the same rule: each per-character image task uses the reference images currently stored for that character, if any exist.

## UX Rules

- Reference images are character-scoped, not project-scoped.
- The feature must never block prompt-only generation.
- The first release supports only:
  - add
  - preview
  - delete
- No ordering controls are shown.
- No extra "use references" toggle is shown.
- The system behavior is:
  - if references exist, use them
  - if references do not exist, generate normally

## Data Model

Add a new shared type for reference images:

```ts
type CharacterReferenceImage = {
  id: string;
  fileName: string;
  originalFileName: string;
  mimeType: string;
  sizeBytes: number;
  createdAt: string;
};
```

Extend the character-sheet detail payload:

```ts
type CharacterSheetRecord = {
  // existing fields...
  referenceImages: CharacterReferenceImage[];
};
```

The existing list response may continue to reuse the same record shape so Studio does not need a second DTO family.

## Storage Layout

Reference images live inside the current character's asset directory:

```text
projects/<project>/character-sheets/batches/<batch>/characters/<character>/
  references/
    manifest.json
    ref-001.png
    ref-002.jpg
```

Rules:

- `manifest.json` is the source of truth for the character's current reference-image set
- each image file name is generated by the backend and does not rely on the browser-provided file name
- `originalFileName` is preserved in the manifest for display only
- deleting a reference image removes both the file and its manifest entry

## API Design

Add the following routes under the existing character-sheet route group:

1. `POST /projects/:projectId/character-sheets/:characterId/reference-images`
   - accepts `multipart/form-data`
   - supports one or more uploaded image files
   - validates image mime type
   - saves files into the character reference directory
   - returns the updated `CharacterSheetRecord`

2. `DELETE /projects/:projectId/character-sheets/:characterId/reference-images/:referenceImageId`
   - deletes the matching file and manifest entry
   - returns the updated `CharacterSheetRecord`

3. `GET /projects/:projectId/character-sheets/:characterId/reference-images/:referenceImageId/content`
   - streams the saved image bytes for browser preview
   - exists only to support Studio thumbnails and refresh-safe previews

The existing `GET /projects/:projectId/character-sheets/:characterId` route returns `referenceImages` so Studio can hydrate character details in one request.

## Service Boundaries

Reference images should stay in filesystem storage, not SQLite.

Recommended split:

- `packages/core`
  - define reference-image DTOs and storage ports
  - expose use cases for add/delete/list-through-detail
  - pass resolved local file paths into task inputs
- `packages/services`
  - implement filesystem storage for manifest and files
  - resolve absolute file paths from character-owned reference assets
- `apps/api`
  - parse multipart uploads
  - expose routes for add/delete/content
- `apps/studio`
  - render the reference-image section
  - upload browser files
  - delete images
  - refresh character detail state

## Generation Wiring

The worker-side image-generation provider already supports:

```ts
referenceImagePaths?: string[];
```

The new integration rule is:

- when building a `character_sheet_generate` task input, populate `referenceImagePaths` from the character's currently stored reference images
- when no stored reference images exist, omit the field or pass an empty array
- `processCharacterSheetGenerateTask` forwards that field unchanged to the turnaround image provider

This preserves current generation behavior while enabling optional reference guidance.

## Error Handling

Upload-time failures:

- invalid mime type
- empty upload payload
- missing character or project
- filesystem write failure

Behavior:

- upload failure affects only the upload action
- do not change the character generation status on upload failure

Delete-time failures:

- missing reference image id
- filesystem delete failure

Behavior:

- delete failure affects only the delete action

Generation-time failures:

- no reference images: continue prompt-only generation
- reference images exist but the external uploader fails: fail that character-generation task using the existing task-failure path

## Studio UI

In the character detail panel, add a `参考图` section with:

- an `添加参考图` file-picker button
- a thumbnail grid
- one delete action per thumbnail

No extra settings are shown in the first release.

The `重新生成` button keeps its current label and placement. Its behavior changes only by automatically including current reference images when available.

## Testing Strategy

### Core

- adding reference images updates the returned character detail
- deleting reference images updates the returned character detail
- regenerate-task creation includes `referenceImagePaths` when references exist
- generate-task creation still works without references

### Services

- storage writes manifest and image files
- storage deletes files and manifest entries
- storage resolves absolute local file paths for generation
- content lookup returns the expected file path and mime type

### API

- upload route accepts multipart image files and returns updated detail
- delete route removes one reference image
- content route returns image bytes
- detail route includes `referenceImages`
- non-image uploads return a validation error

### Studio

- character detail renders existing reference images
- uploading files calls the new API and refreshes state
- deleting a reference image calls the new API and refreshes state
- regenerate still works when `referenceImages` is empty

## Risks

- multipart parsing adds a new API dependency and route behavior that is not currently present
- preview URLs need a stable authenticated-enough path model for browser rendering
- file-backed manifests can drift if writes are not done atomically enough; first release should rewrite the full manifest after each successful change

## Decision Summary

- persist reference images per character inside project storage
- keep reference images optional for generation
- support preview and delete only in the first release
- reuse existing worker-side VectorEngine reference-image upload support
- expose content through API routes instead of browser-side direct file access
